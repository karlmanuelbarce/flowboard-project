## Services Architecture

| Service | Technology              | Port | Responsibility                                              |
|---------|-------------------------|------|-------------------------------------------------------------|
| api     | Node.js / Express / TS  | 3000 | All HTTP requests; input validation; business logic; events |
| worker  | Node.js / TS            | —    | Redis Stream consumer; writes AuditLog; dead-letter handling |
| db      | PostgreSQL 16           | 5432 | Primary data store                                          |
| redis   | Redis 7                 | 6379 | Rate limiting, refresh tokens, event stream                 |
| nginx   | Nginx Alpine            | 80   | Reverse proxy → api:3000                                    |

**Request flow for PATCH /tasks/:id:**
1. Request hits Nginx on port 80
2. Nginx proxies to api container on port 3000
3. `authenticate` middleware validates JWT (no Redis/DB call — JWT is self-contained)
4. Route handler validates `req.params` and `req.body` with Zod
5. Prisma Client updates task in PostgreSQL
6. API publishes `TASK_UPDATED` event to `tasks:events` Redis Stream
7. API returns `{ success: true, data: updatedTask }` to client
8. Worker reads the event, writes AuditLog row, calls XACK

## Error Handling

- All async route handlers use `try/catch`
- Errors are always passed to `next(err)` — never `res.status().json()` inside a catch block
- HTTP errors use the `AppError` class: `throw new AppError('Task not found', 404, 'TASK_NOT_FOUND')`
- The global error handler is the last middleware in `app.ts`
- In `NODE_ENV=production`: no stack traces in error responses
- Non-AppError errors (Prisma errors, TypeErrors) are caught by the global handler and returned as 500

```ts
// AppError usage
throw new AppError('message', httpStatusCode, 'ERROR_CODE_STRING');

// Global handler distinguishes AppError from unknown errors:
// - AppError → use statusCode + code from the instance
// - ZodError → 422 Unprocessable Entity
// - Unknown → 500 Internal Server Error (no stack trace in production)
```

## TypeScript Rules

- Strict mode is on — zero `implicit any` tolerated
- All function parameters and return types must be annotated
- Prefer `interface` for object shapes; `type` for unions and intersections
- Use `z.infer<typeof Schema>` for request body/param types — never duplicate type definitions
- Route handlers must be typed: `Request<Params, ResBody, ReqBody>`, return `Promise<void>`
- No `as any` casts — fix the type properly

```ts
// Correct pattern for a typed route handler
export const getTask = async (
  req: Request<z.infer<typeof TaskIdParam>>,
  res: Response,
  next: NextFunction
): Promise<void> => {
  try {
    const { id } = TaskIdParam.parse(req.params);
    // ...
  } catch (err) {
    next(err);
  }
};
```

## Response Format

All responses follow this exact shape:

```json
// Success
{ "success": true, "data": { ... } }

// Error
{ "success": false, "message": "Human-readable message", "code": "SCREAMING_SNAKE_CODE" }
```

- `201 Created` for POST that creates a resource
- `204 No Content` for DELETE (no body)
- `401 Unauthorized` for missing/invalid token
- `403 Forbidden` for valid token but wrong ownership
- `404 Not Found` for missing resource
- `422 Unprocessable Entity` for Zod validation failures
- `429 Too Many Requests` for rate limit exceeded

## Input Validation Rules

- Every route that accepts `req.body` or `req.params` has a Zod schema
- Path parameter IDs are validated as UUIDs: `z.string().uuid()`
- **Never** spread `req.body` directly into Prisma — always parse through Zod first
- Unknown/extra fields in request body are stripped (use `z.object({}).strict()` or Zod's default strip behavior)
- ZodError is caught by the global error handler and returned as 422

```ts
const CreateTaskSchema = z.object({
  title: z.string().min(1).max(255),
  description: z.string().optional(),
  priority: z.enum(['LOW', 'MEDIUM', 'HIGH']).default('MEDIUM'),
  boardId: z.string().uuid(),
});
type CreateTaskInput = z.infer<typeof CreateTaskSchema>;
```

## Authentication & Authorization

- Access tokens: JWT, 15-minute TTL, signed with `JWT_SECRET`
- Refresh tokens: JWT, 7-day TTL, stored in Redis, invalidated on use
- Redis key for refresh tokens: `refresh:{userId}:{tokenId}`
- `authenticate` middleware: verifies JWT, attaches `req.user = { id, email }` to request
- All `/boards` and `/tasks` routes require a valid Bearer token
- Ownership check: `resource.ownerId === req.user.id` — return 403 if mismatch
- Token rotation: on `/auth/refresh`, delete old token from Redis before issuing new pair
- Replay attack: second use of the same refresh token returns 401

## Redis Key Conventions

| Key Pattern                    | Type        | TTL       | Purpose                            |
|--------------------------------|-------------|-----------|------------------------------------|
| `rate:{ip}`                    | String      | 900s      | Rate limiter counter per IP        |
| `refresh:{userId}:{tokenId}`   | String      | 604800s   | Refresh token store (7 days)       |
| `tasks:events`                 | Stream      | —         | Task event stream (Redis Streams)  |
| `tasks:events:dlq`             | List        | —         | Dead-letter queue for failed events |

**Dead-Letter Queue (DLQ) format:**

Key: `tasks:events:dlq`
Type: Redis List (LPUSH)
Entry format: JSON string —
```json
{
  "streamId": "1234567890123-0",
  "action": "TASK_UPDATED",
  "taskId": "uuid",
  "userId": "uuid",
  "payload": {},
  "ts": "1234567890123",
  "failedAt": "ISO-8601 timestamp",
  "retries": 3,
  "lastError": "Error message string"
}
```
Events are moved to DLQ after **3 failed processing attempts**. They are never dropped silently.

## Data Model (Prisma Schema)

```prisma
model User {
  id        String     @id @default(uuid())

  password  String

  id        String   @id @default(uuid())

  id          String     @id @default(uuid())
  title       String
  description String?

  id        String   @id @default(uuid())
  userId    String

  action    String
  entity    String
  entityId  String
  createdAt DateTime @default(now())
}

enum TaskStatus { TODO IN_PROGRESS REVIEW DONE }
enum Priority   { LOW MEDIUM HIGH }
```

## Security Middleware Stack (order matters in app.ts)

```ts
app.use(helmet())
app.use(cors({ origin: process.env.ALLOWED_ORIGINS?.split(','), credentials: true }))
app.use(express.json({ limit: '10kb' }))
app.set('trust proxy', 1)
// pino-http request logger
// routes
// globalErrorHandler  ← must be last
```

- Helmet sets security headers on all responses
- CORS allows only origins listed in `ALLOWED_ORIGINS` env var
- Body size limit: 10kb — returns 413 for larger payloads
- Rate limiter applied only to `POST /auth/login` and `POST /auth/register`
- Rate limiter fails **open** if Redis is down (log the error, do not block traffic)

## Worker — Redis Stream Consumer

- Consumer group: `audit-group`
- Consumer name: `worker-1`
- Reads from: `tasks:events` using `XREADGROUP`
- Batch size: 10 messages per read
- On success: write AuditLog to PostgreSQL, then call `XACK`
- On failure: retry up to 3 times; on 3rd failure, push to `tasks:events:dlq` (see DLQ format above)
- Worker has no HTTP server — it is a long-running loop only

## Testing Conventions

- Tests run via: `docker compose -f docker-compose.test.yml exec api npx jest`
- Coverage check: `docker compose exec api npx jest --coverage` — must hit **≥ 80% lines and branches**
- Every endpoint has at minimum: 1 happy-path test + 1 error-path test
- Test setup: register a user in `beforeAll`, use the returned `accessToken` for authenticated calls
- Test teardown: call `prisma.$disconnect()` in `afterAll`
